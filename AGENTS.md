# Learn NestJS 仓库协作规范

本文件用于约束在本仓库中新增、修改和审查学习内容时的行为。所有课程文档、Demo、测试和导航调整都应遵循以下规则。

## 1. 项目定位

- 本仓库是一个 NestJS 学习知识库，不是某个业务系统的生产代码仓库。
- 目标读者是具有约 6 年前端开发经验、熟悉 TypeScript 和常见前端工程化工具、希望转向高级全栈开发的工程师。
- 主课程固定为 5 天、15 课，每天 3 课。内容强调快速建立完整后端能力，不追求覆盖 NestJS 的每一个 API。
- 课程以知识管理 API 为贯穿项目；每节课都要同时回答：
  1. 这个能力解决什么问题？
  2. 它位于 NestJS 或请求链路的什么位置？
  3. 它在真实项目中应当怎样使用？
- 优先讲清架构边界、工程取舍和常见错误，避免面向初学者反复解释 TypeScript、npm、Promise 等基础知识。

## 2. 学习文档写作规则

- 每课内容应能在 60–90 分钟内读完并跑通，文字保持精炼，以“理解核心概念 + 运行 Demo + 观察实际行为”为主。
- 学习文档站在开发者解决问题和记录实践经验的角度编写，不使用讲师授课、布置作业或考核学习者的口吻。
- 优先使用“遇到什么问题、代码如何工作、怎样运行验证、真实项目中如何取舍”等表达，不使用“完成本课后你应该”“请回答以下问题”等课堂话术。
- 一课只设一个清晰主题；后续课程的知识可以点到为止，但不要提前大篇幅展开。
- 新概念按照以下顺序组织：问题场景 → 核心心智模型 → 最小代码 → 运行观察 → 工程实践与易错点。
- 可以使用前端开发中的相似概念帮助迁移理解，但必须指出差异，不能把类比当作定义。
- 代码示例应来自本课 Demo 或能直接对应到 Demo，避免文档代码与实际项目脱节。
- 对存在多种实现方案的内容，应先给出课程采用的方案和原因，再简要说明替代方案及适用边界。
- 明确区分当前主题的核心内容、延伸内容和后续课程依赖，避免无边界扩展。
- 不设置“自测”“完成标准”“课后作业”等教学或考核式固定模块。必要的验证步骤直接放在相关代码和运行说明之后。
- 每个章节都应为理解或使用 Demo 服务；不能提供实际价值的总结、清单和重复说明应删除。

## 3. 课程目录和文档约束

每个 `lessons/<序号>-<主题>/` 目录必须包含：

```text
lessons/<序号>-<主题>/
├── index.md       # 中文学习文档，课程内容的事实来源
├── index.en.md    # 与中文文档结构和含义一致的英文版本
└── demo/          # 本课可独立运行的完整 NestJS 项目
```

- 中文文档更新后必须同步更新英文文档；两者的标题、主要章节、代码示例和运行说明应保持一致。
- 中文文档优先保证自然、准确和易读；英文文档应按含义翻译，不做逐字直译。
- 每课文档围绕问题背景、核心内容、Demo 运行与验证、工程实践和易错点组织，不强制凑齐固定模块。
- 标题和目录名使用既有编号与 kebab-case 命名，不随意调整课程顺序。
- 跨课程导航统一维护在 `lessons/README.md`。
- 仓库介绍、安装方式和通用参考资料统一维护在根目录 `README.md`。
- 课程末尾不设置重复的“参考资料”模块。确实只针对某个知识点的资料，可以在相关正文中就近链接。
- 不在多个课程中复制大段相同理论；需要回顾时用一小段摘要并链接到前置课程。

## 4. 图示与可视化规则

- 图比大段文字更适合表达结构和关系。复杂或重要的 NestJS 逻辑只要能够通过图清晰表达，就必须配图，不能只依赖连续文字或 ASCII 图。
- 所有技术图统一使用仓库内的 `baoyu-diagram` skill 创建，不使用 Mermaid、Excalidraw 文件、截图或临时手绘图替代。图示采用该 skill 规定的浅色白板风格，并输出可直接嵌入文档的 SVG。
- 以下内容默认需要评估并优先配图：
  - 模块层级、依赖注入关系和应用架构；
  - 请求生命周期、认证授权、Guard/Pipe/Interceptor/Filter 执行顺序；
  - 跨组件时序、事件流、消息队列和后台任务；
  - 状态变化、事务、并发、缓存和幂等分支；
  - 数据流、部署拓扑、CI/CD 和故障处理路径。
- 出现三个及以上相互关联的组件、多个连续阶段、条件分支、状态迁移或难以线性描述的层级时，应使用图示。
- 简单命令、单个 API、两项对比或可用一小段代码说明的内容不强制配图，避免装饰性插图。
- 图用于承担主要的结构表达，正文只补充关键结论、边界和工程取舍，不再逐项重复图中全部内容。
- 每张图只保留一个自包含、可编辑的 SVG，不生成或提交 PNG、JPEG、PDF 等伴随文件。
- 课程图统一存放在 `lessons/<课程目录>/diagram/`。中文图使用 `<topic>.svg`，英文图使用 `<topic>.en.svg`；没有文字或语言无关时可以共用。
- 中文和英文文档中的图示逻辑必须一致。若图中包含自然语言标签，应分别生成中英文 SVG。
- 使用 `baoyu-diagram` 时必须先读取与图类型对应的 reference，按其布局、间距和验证流程生成，并检查文字溢出、元素重叠、箭头方向和 viewBox。

## 5. Demo 的渐进规则

- 每课的 `demo/` 都必须是完整、可独立启动的 NestJS 项目，不能只存放零散代码片段。
- Demo 采用累计快照：第 N 课必须包含前 N-1 课已经实现的能力，并在此基础上增加本课内容。
- 第 15 课 Demo 必须包含整套课程的累计业务逻辑和工程能力。
- 累计不等于无条件复制错误。修复前置逻辑时，应同步评估并修正受影响的后续 Demo。
- 每课新增的核心知识必须至少有一个可观察结果，例如 HTTP 接口、日志、测试、任务执行结果或部署配置。
- 每个 Demo 必须有自己的 `README.md`，说明启动命令、演示步骤、预期结果和本课边界。
- 每个 Demo 的默认端口使用 `3000 + 课程序号`，即第 1 课为 `3001`，第 15 课为 `3015`。
- 环境变量应提供 `.env.example`；示例中不得包含真实密钥、Token、账号或生产地址。
- 外部服务应说明启动方式。若课程已有内存降级方案，应保证没有 Docker、Redis 等外部依赖时仍能完成基础学习流程。
- 不使用符号链接或隐式引用其他课程源码来伪装“独立项目”；进入任意 Demo 目录都应能理解其完整结构。

## 6. NestJS 代码约束

- 使用 TypeScript 严格模式，避免 `any`、无意义的类型断言和未处理的 Promise。
- 优先使用 NestJS 的模块、Provider、依赖注入和平台无关 API；仅在确有平台特有需求时直接使用 Express/Fastify 对象。
- Controller 负责协议适配和输入输出，业务逻辑放入 Service/Provider，不在 Controller 中堆积业务规则。
- 输入模型使用 DTO；涉及外部输入时应根据课程进度加入校验、转换和明确的错误响应。
- 模块边界通过 `imports`、`providers` 和 `exports` 清晰表达，避免随意创建全局模块或循环依赖。
- 配置通过环境变量和配置模块注入，不在源码中硬编码凭据或环境相关值。
- API、DTO、源码和学习文档中的示例必须保持一致；修改接口时同步检查所有引用它的课程内容。
- 依赖版本以仓库根目录及 Demo 的 `package.json` 为准。新增依赖前先确认 NestJS 主版本和现有工具链兼容。
- 不提交 `node_modules/`、`dist/`、`coverage/`、本地 `.env`、数据库临时文件或运行日志。

## 7. 验证规则

修改课程 Demo 后，至少在对应 workspace 中执行：

```bash
npm run lint --workspace <demo-workspace-name>
npm run build --workspace <demo-workspace-name>
```

- 除第 13 课“自动化测试”外，其他课程的 Demo 不编写或保留单元测试、集成测试、E2E 测试及测试目录；验证以 Lint、构建和 `demo/README.md` 中可直接执行的本地运行步骤为准。
- 第 13 课专门讲自动化测试，必须包含单元、集成或 E2E 示例，并运行该课文档要求的相关测试命令。第 14、15 课累计业务源码，但不复制第 13 课测试文件。
- 修改累计逻辑时，除了当前课程，还要验证所有直接受影响的后续课程；至少抽查最终第 15 课 Demo。
- 学习文档修改后检查中英文结构是否同步、命令是否可执行、路径和端口是否与 Demo 一致。
- 提交前执行 `git diff --check`，确保没有空白符错误；不要为了本次任务格式化或改写无关文件。
- 如果受环境限制无法完成某项验证，必须明确记录未验证的项目和原因，不能默认其已通过。

## 8. 内容归属规则

- `README.md`：项目定位、全局安装/运行方式、通用参考资料和仓库级导航。
- `lessons/README.md`：5 天 15 课的顺序、每日主题和课程入口。
- `lessons/*/index.md`：以开发者视角记录本课的问题、实现、验证方式和工程取舍。
- `lessons/*/demo/README.md`：只记录对应 Demo 的运行、演示和验证方式。
- 可复用的通用规则应提升到仓库级文档，不在每课末尾重复维护。

## 9. 变更检查清单

完成一项课程变更前，确认：

- [ ] 内容符合 5 天速学和高级前端转全栈的目标；
- [ ] 没有引入与本课无关的过量知识；
- [ ] 中文和英文学习文档已经同步；
- [ ] 文档采用开发者视角，没有讲师式提问、作业或考核模块；
- [ ] 复杂或重要逻辑已经使用 `baoyu-diagram` 配图，图与正文及 Demo 一致；
- [ ] 文档中的路径、命令、端口和响应与 Demo 一致；
- [ ] Demo 可以独立运行，并保留了此前课程的累计能力；
- [ ] 新知识有可观察的本地运行结果；仅第 13 课包含测试用例；
- [ ] Lint 和构建已经通过；第 13 课的相关测试也已经通过；
- [ ] 通用资料没有重复放入课程末尾；
- [ ] 没有提交秘密信息、构建产物或无关改动。

## 10. 课程完成后的 Reviewer 门禁

- 每当主 Agent 完成一节课程的新增或修改，在向用户交付前必须生成一个 `reviewer` 自定义子 Agent，对该课程执行独立审查。
- `reviewer` 的项目级配置位于 `.codex/agents/reviewer.toml`，使用当前环境可用的轻量 GPT-5.6 Terra；主 Agent 不得用普通 Agent 或自审替代。第 1–3 课已完成审查，不再重复触发；门禁从第 4 课继续。
- 触发时必须向 reviewer 提供本次完成的课程目录、变更范围、已执行的验证及其结果，并要求它读取实际文件和 diff 后再给结论。
- reviewer 以 `lessons/README.md` 为课程大纲的主基准，同时使用根目录 `README.md` 和本文件作为仓库级约束。大纲、文档和实现之间发生冲突时必须报告，不得静默选择其中一方。
- 审查至少覆盖：课程大纲是否有遗漏，中文和英文课程文档是否正确且同步，文档与 Demo 是否一致，示例源码是否正确，累计 Demo 是否受影响，以及图示、验证和仓库卫生是否符合规则。只有第 13 课审查测试用例；其他课程若包含测试文件应作为规则错误报告。
- 这是阻断式门禁。reviewer 报告任何确认问题或无法完成的必要验证后，主 Agent 必须修复或补齐验证，然后重新生成 reviewer 复审；不得沿用先前 reviewer 实例自行认可修复。
- 只有最新一轮 reviewer 明确返回 `REVIEW PASS`，主 Agent 才能宣布该课程完成。不得把未解决问题降级为建议后交付。
- reviewer 只读审查，不直接修改课程文件；所有修复由主 Agent 完成，以保持审查独立性。
